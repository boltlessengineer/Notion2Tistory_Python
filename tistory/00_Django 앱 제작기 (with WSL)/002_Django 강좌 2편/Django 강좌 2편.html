Django 프로젝트 생성

우선 이전에 만들었던 가상환경이 활성화된 상태인지(터미널에서 가상환경에 접속 중인지) 확인한다. 

가상환경이 활성화된 상태라면 Django 프로젝트의 폴더를 생성하고 들어간다. 

mkdir my_django_proj

cd my_django_proj

my_django_proj 이라는 폴더를 생성하고 ./my_django_proj 경로로 이동했다. 

 

django-admin startproject config .

django-admin startproject 명령어를 이용해 Django 프로젝트를 만들어 줄 수 있다. 앞서 my_django_proj 이라는 폴더를 생성했으므로 config 다음에 . 을 붙여줘서 my_django_proj 폴더를 Django 프로젝트 폴더로 만들 수 있다. 

MDN에서는 config . 식으로 치지 않고 locallibrary 식으로 입력하던데 그러면 locallibrary 폴더 안에 locallibrary 폴더가 생성된다. 나는 위키독스의 점프 투 장고에서 쓴 방식대로 my_django_proj 폴더를 Django 프로젝트 폴더로 만들고. 포로젝트 관련 파일들을 config 폴더 안에 두도록 했다. 

 

tree 명령어를 사용하면 아래와 같이 my_django_proj 폴더 안에 아래와 같이 파일들이 생성된 것을 볼 수 있다. 

tree 명령어는 apt install tree 로 받을 수 있다. 

.
├── manage.py
└── config
    ├── __init__.py
    ├── asgi.py
    ├── settings.py
    ├── urls.py
    └── wsgi.py

 

거듭 말하지만 이건 가상환경에 접속된 상태에서 생성한 Django 프로젝트의 폴더이다. 각각의 Django 앱은 이곳의 하위 디렉터리가 된다.

이렇게 프로젝트-앱 구조로 관리하는 이유는 Django가 기본적으로 백엔드 특화이기 때문인 것 같다. 하나의 큰 프로젝트가 있을 때, 여러가지 백엔드 앱이 있을 텐데, 프론트엔드와 달리 백엔드는 하나의 DB를 공유하거나 하는 등의 상호작용이 이뤄져야 하기 때문이다. 

설정 변경하기

실행 전에 Django 의 설정을 조금 바꿔줘야 할 필요가 있다. (필수는 아님) 

/config/ 폴더의 settings.py 파일에서 Django 프로젝트 설정을 수정해 줄 것이다. 

시간대 및 언어

LANGUAGE_CODE = 'en-us'

TIME_ZONE = 'UTC'

👆위 두줄의 코드를 찾아서 👇아래와 같이 바꾼다. 

LANGUAGE_CODE = 'ko'

TIME_ZONE = 'Asia/Seoul'

 

LANGAUGE_CODE 는 이후 다루게 될 Django 관리자 페이지에 표시되는 언어를 말한다. 

TIME_ZONE 는 말 그대로 웹사이트의 시간대를 설정하는 것이다. 위키피디아 타임존 리스트에서 알맞은 시간대를 선택하여 넣어줄 수 있다. 한국은 Asia/Seoul 이면 된다. 

정적 파일 경로

웹사이트에서 사용되는 아이콘, 배경 이미지등 정적인 파일들을 찾을 경로를 설정해주자 

STATIC_ROOT = os.path.join(BASE_DIR, 'static')

settings.py 파일 제일 아래 줄, STATIC_URL = '/static/' 코드 아래에 위 코드를 넣어준다. 

Django 프로젝트 실행

계속 코드만 작성하는 것은 재미 없으니 여기서 잠깐 my_django_porj 프로젝트를 실행해보자 

가상환경에 접속된 상태에서 터미널에 아래와 같이 입력한다. 

python manage.py runserver

 

웹 브라우저에 "http://127.0.0.1:8000/" 또는 "http://localhost:8000"을 입력하면 아래와 같이 프로젝트가 실행되는 것을 볼 수 있다. 

 

Django 앱 생성

이제 Django 프로젝트 안에 Django 앱을 생성해주자. 

 

python3 manage.py startapp dboard

manage.py 가 위치한 경로에서 위와 같이 입력하면 dboard 라는 Django 앱이 생성된다. 

 

다시 tree 명령어를 입력하여 폴더 구조를 보면 이렇게 바뀐 것을 알 수 있다. 

.
├── config
│   ├── __init__.py
│   ├── __pycache__
│   │   ├── __init__.cpython-38.pyc
│   │   └── settings.cpython-38.pyc
│   ├── asgi.py
│   ├── settings.py
│   ├── urls.py
│   └── wsgi.py
├── dboard
│   ├── __init__.py
│   ├── admin.py
│   ├── apps.py
│   ├── migrations
│   │   └── __init__.py
│   ├── models.py
│   ├── tests.py
│   └── views.py
└── manage.py

 

 

  • __init__.py : 빈 파일. Python에게 이 디렉토리를 하나의 Python 패키지로 다루도록 지시
  • settings.py : 웹사이트의 설정. 프로젝트의 application이 등록되는 곳이며, static files의 위치, database 세부 설정 등이 들어감.
  • urls.py : 사이트의 url과 view를 연결해줌. React의 react-router 의 Router.js 파일과 유사
  • wsgi.py : 장고 어플리케이션이 웹서버와 연결 및 소통하는 것을 도움
  • manage.py : 어플리케이션을 생성하고, 데이터베이스와 작업하고, 그리고 개발 웹 서버를 시작하기 위해 사용됨
  • migrations/ : 모델을 수정할 때 마다 자동으로 데이터베이스를 업데이트하는 것을 가능하게 해줄 파일들을 저장하는 폴더

 

기타 모듈 종류

아래는 애플리케이션 폴더에 추가로 넣어줄 수 있는 모듈에 대해 Django 개발자들끼리 정한 이름 규칙 약속이다. (퍼옴) 

  • 목록
    • forms.py: 입력 폼 선언
    • behaviors.py: 모델 믹스인 위치에 대한 옵션
    • constants.py: 앱에 쓰이는 상수 선언
    • decorators.py: 데코레이터
    • db/: 여러 프로젝트에서 사용되는 커스텀 모델이나 컴포넌트
    • fields.py: 폼 필드
    • factories.py: 테스트 데이터 팩토리 파일
    • helpers.py: 뷰와 모델 파일을 가볍게 하기 위해 유틸리티 함수 선언
    • managers.py: models.py가 너무 커질 경우 커스텀 모델 매니저가 위치
    • signals.py: 커스텀 시그널
    • viewmixins.py: 뷰 모듈과 패키지를 더 가볍게하기 위해 뷰 믹스인을 이 모듈로 이전

 

앱 등록하기

바로 위에서 언급한 settings.py 파일, 설명 그대로 프로젝트에 앱을 만들었으면 settins.py 파일에 등록해줘야 한다. 

settings.py 파일을 열고, INSTALLED_APPS 배열을 찾는다. 

INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',
]

위와 같은 상태일텐데, 👇배열 끝에 생성한 앱(dboard)을 등록해준다. 

INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',
    'dboard',
]

DB 설정

이 부분은 건드릴 필요가 없다. Django는 기본적으로 sqlite3 Database가 지정되어있다. 

settings.py 파일의 DATABASES 객체를 보면 sqlite3 연동이 기본 설정되어있는 것을 확인 할 수 있다. 

DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.sqlite3',
        'NAME': os.path.join(BASE_DIR, 'db.sqlite3'),
    }
}

나는 가능하면 한두번 다뤄봤던 mySQL으로 해보고 싶지만 넘어가자 

URL 맵핑

리액트에서 url 매핑을 위해 react-router를 import해서 Router.js 파일을 만들듯이, 장고는 urls.py에서 url 맵핑을 한다. 

./config/urls.py 파일을 열어보면 아래와 같이 되어 있는 것을 알 수 있다. 

from django.contrib import admin
from django.urls import path

urlpatterns = [
    path('admin/', admin.site.urls),
]

urlpatterns 배열 안에 있는 path() 함수로 url과 뷰(이후 설명한다)를 연결해주는 방식이다. 

React랑 겁나 비슷하네 웹 프레임워크는 다 이런가 

 

코드를 보면 admin/ 이라는 경로가 이미 존재하는데, 이 경로에 대해서도 나중에 설명하겠다. 

아래와 같이 바꿔주자. 

from django.contrib import admin
from django.urls import path
from dboard import views

urlpatterns = [
    path('admin/', admin.site.urls),
		path('dboard/', views.index),
]

이제 사용자가 http://localhost:8000/dboard 로 접근하면 dboard 폴더의 views.py 에서 index 함수를 가져와 실행한다. 

그런데 아직 views.py 파일에 index 함수를 만들지 않았으므로 위 경로로 접속해도 아무것도 출력되지 않는다.  

뷰(view) 생성

이제 Django가 /dboard 경로에서 보여줄 index 뷰를 만들어주자. 

뷰(view)는 애플리케이션의 "로직"을 넣는 곳이다. 특정 코드를 처리하고, 템플릿(페이지)를 출력하는, 이번에도 React에 비유해 보자면 React의 Container-Presenter 패턴에서 Container가 뷰에 해당한다고 보면 될 것 같다. 

./dboard/views.py 파일을 연다. 

from django.shortcuts import render

# Create your views here.

👇아래와 같이 index 뷰를 만들어주자 

# from django.shortcuts import render
from django.http import HttpResponse


def index(request):
    return HttpResponse("<h1>Hello World!</h1>")

처음에 from django.shortcuts import render 은 아직 render() 함수는 쓰지 않을 것으므로 일단 주석처리해준다. 

 

이제 http://localhost:8000/dboard 로 접속하면 아래와 같이 출력되는 것을 볼 수 있다. 

 

못한 설명을 마저 하자면... 파이썬의 뷰는 그냥 함수이다. 

  1. 브라우저에서 /dboard 경로로 접근하면
  1. urls.py 의 path() 함수가 ./dboard/views.py 파일의 index() 함수를 호출하고
  1. index() 함수가 로직 코드를 실행하고 최종적으로 다른 경로로 보내거나 페이지를 출력하거나 하는 일을 한다.
  1. HttpResponse() 함수는 html을 그대로 페이지에 보여준다.

대충 이렇게 작동한다고 보면 된다. 

 

위 구조의 HttpResponse() 함수 만으로도 f-string 등을 이용하면 아래와 같이 로직을 처리하고 출력할 수는 있다. 

# from django.shortcuts import render
from django.http import HttpResponse


def index(request):
    return HttpResponse(f"<h1> Hello World!</h1><p>Did you know that? 1 + 1 is {1+1}</p>")

 

하지만 로직 함수 안에 페이지 HTML 요소들을 다 구겨넣는 것은 그다지 아름답지 못하다. 

그런 프론트엔드적 요소를 분리하기 위해 Django는 template 라는 기능을 지원한다. 간단하게 설명하자면 HTML 파일에서 가볍게 파이썬 코드를 실행하여 변수나 배열을 출력하는 것이다.
